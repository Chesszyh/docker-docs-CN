本页详细介绍了 Docker 的产品发布生命周期以及 Docker 如何定义每个阶段。它还提供了关于产品退役流程的信息。功能和产品可能会经历其中的部分或全部阶段。

## 生命周期阶段

| 生命周期阶段 | 客户可用性 | 支持可用性 | 限制 | 退役 |
| --- | --- | ---- | ---| ---|
|[实验性 (Experimental)](#实验性-experimental)| 限制范围可用 | 社区支持 | 软件可能存在限制、Bug 和/或稳定性问题 | 可在不另行通知的情况下停止提供 |
|[测试版 (Beta)](#测试版-beta) | 全部用户或参与测试反馈计划的用户 | 社区支持 | 软件可能存在限制、Bug 和/或稳定性问题 | 可在不另行通知的情况下停止提供 |
| [早期体验 (Early Access, EA)](#早期体验-ea) | 全部用户或参与早期体验反馈计划的用户 | 全面支持 | 软件可能存在限制、Bug 和/或稳定性问题。这些限制将被文档化。 | 遵循 [退役流程](#退役流程) |
| [正式发布 (General Availability, GA)](#正式发布-ga) | 全部用户 | 全面支持 | 对于受支持的用例几乎没有或完全没有限制 | 遵循 [退役流程](#退役流程) |

### 实验性 (Experimental)

实验性产品是 Docker 目前正在尝试的功能。访问实验性功能的客户有机会对未来的功能进行测试、验证并提供反馈。这有助于我们专注于为客户提供最有价值的工作。

**客户可用性**：实验性功能的可用性是有限的。一部分用户可能无法访问任何实验性功能，或者只能访问其中一个或多个。

**支持**：对实验性功能的支持通过社区支持渠道和论坛以“尽力而为”的方式提供。

**限制**：实验性功能可能存在潜在的重大限制，例如功能限制、性能限制和 API 限制。功能和编程接口可能随时更改，恕不另行通知。

**退役**：在实验阶段，Docker 将决定是否继续推动该产品进入下一个生命周期。我们保留随时更改实验性产品或功能范围或停止提供的权利，恕不另行通知，详见我们的 [订阅服务协议](https://www.docker.com/legal/docker-subscription-service-agreement)。

### 测试版 (Beta)

测试版产品是未来潜在产品或功能的初始发布版本。参与我们测试计划的客户有机会对未来的功能进行测试、验证并提供反馈。这有助于我们专注于为客户提供最有价值的工作。

**客户可用性**：通过邀请或使用产品中明确标识的测试版功能来参与测试版发布。测试邀请可能是公开的，也可能是私有的。

**支持**：对测试版功能的支持通过社区支持渠道和论坛以“尽力而为”的方式提供。

**限制**：测试版发布可能存在潜在的重大限制，例如功能限制、性能限制和 API 限制。功能和编程接口可能随时更改，恕不另行通知。

**退役**：在测试阶段，Docker 将决定是否继续推动该产品进入下一个生命周期。我们保留随时更改测试版产品或功能范围或停止提供的权利，恕不另行通知，详见我们的 [订阅服务协议](https://www.docker.com/legal/docker-subscription-service-agreement)。

### 早期体验 (EA)

早期体验产品或功能可能存在潜在的功能限制，并作为增量推出策略的一部分为特定用户群组启用。它们已经准备好向全球发布，但仍需进行一些微调。

**客户可用性**：早期体验功能可以向所有客户或特定用户群体推出，作为现有功能的补充或替代。

**支持**：早期体验产品提供与正式发布（GA）功能和产品相同级别的支持。

**限制**：早期体验版本可能存在潜在的重大限制，如功能限制、性能限制和 API 限制，不过这些限制将被文档化。对功能和编程接口的重大变更将遵循下文所述的 [退役流程](#退役流程)。

**退役**：如果我们要在正式发布前退役某个早期体验产品，我们将努力遵循下文所述的 [退役流程](#退役流程)。

### 正式发布 (GA)

正式发布产品是功能齐全的产品或功能，可供所有 Docker 客户公开访问。

**客户可用性**：所有 Docker 用户均可根据其订阅级别访问 GA 产品。

**限制**：正式发布的功能和产品对于受支持的用例几乎没有或完全没有限制。

**支持**：所有 GA 产品均获得全面支持，详见我们的 [支持页面](https://www.docker.com/support/)。

**退役**：正式发布产品遵循下文所述的 [退役流程](#退役流程)。


## 退役流程

决定退役或弃用某项功能遵循严格的流程，包括了解需求、使用情况、功能退役的影响，以及最重要的客户反馈。我们的目标是将资源投入到能为大多数客户增加最大价值的领域。

Docker 致力于在与客户互动时保持清晰、透明和主动，尤其是关于我们平台的变更。为此，我们在退役功能时将尽最大努力遵循以下指南：

- **提前通知**：对于主要功能或产品的退役，我们将尝试至少提前 6 个月通知客户。
- **可行的替代方案**：Docker 在退役功能时将努力为客户提供可行的替代方案。这些可能是来自 Docker 的替代产品，也可能是来自第三方供应商的推荐替代方案。在可能且适当的情况下，Docker 会自动将客户迁移到退役功能的替代方案中。
- **持续支持**：Docker 承诺在功能退役日期之前继续提供支持。

在特殊情况下，我们可能需要加速功能退役的时间表，例如为了保护我们平台的完整性或客户及他人的安全而必须进行的必要更改。在这些情况下，尽快进行此类更改至关重要。

同样，由于第三方决定更改或退役其解决方案，集成的第三方软件或服务可能也需要退役。在这种情况下，退役的节奏将不受我们控制。

然而，即使在这些情况下，我们也将在可能的情况下提供尽可能早的提前通知。
