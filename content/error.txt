这是一个典型的 Node.js 内存溢出（Out of Memory, OOM） 错误。具体到 gemini-cli 的上下文中，这意味着 Node.js 进程在执行任务时耗尽了分配给它的堆内存（Heap Memory）。

  错误核心原因
   1. 内存达到限制：报错显示堆内存已经达到了约 4030 MB（约 4GB）。在 64 位系统上，Node.js 默认的内存限制通常就在 4GB 左右。
   2. 垃圾回收失效：Ineffective mark-compacts near heap limit 表示 Node.js 试图通过垃圾回收（GC）释放内存，但回收效率极低（回收后内存依然接近上限），无法满足后续的内存分配需求，因此直接崩溃。

  为什么在 gemini-cli 中会发生？
  在你刚才执行的文档中文化任务中，可能有以下诱因：
   * 处理文件过多：一次性扫描、读取并处理了 guides/ 目录下的大量 Markdown 文件。
   * 上下文累积：CLI 在运行过程中需要维护对话历史、文件结构和翻译缓存，如果处理的文件非常多，内存占用会持续升高。
   * 大文件读取：如果有单个 Markdown 文件体积特别大，或者包含复杂的结构，处理时会产生大量中间对象。

  解决方案

  1. 临时增加 Node.js 内存限制
  你可以通过环境变量增加 Node.js 的最大堆空间。例如，将其提升到 8GB：

   1 export NODE_OPTIONS="--max-old-space-size=8192"
  然后再运行 gemini-cli。

  2. 分批处理任务（推荐）
  不要一次性翻译整个 guides/ 目录。你可以按子目录分批进行：
   * 先处理 guides/rust/
   * 再处理 guides/reactjs/
   * 以此类推

  3. 清理不需要的资源
  如果你在 CLI 中运行了很长时间，可以尝试重启 CLI 进程，以释放由于长时间运行累积的内存占用。

  4. 检查是否有超大文件
  检查 guides/ 目录下是否有体积异常巨大的 .md 文件。如果有，可以先跳过或单独处理。

  ---

  建议：由于你刚才已经翻译了大部分内容，如果现在重启 CLI，由于我已经完成了大部分文件的写入，接下来的扫描会很快跳过已存在的文件，内存压力应该会减小。你可以尝试重启并继续执行未完成的部分。
