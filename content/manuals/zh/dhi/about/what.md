在当今多样化的软件环境中，容器镜像通常设计为具有灵活性和广泛的兼容性。虽然这使它们适用于许多用例，但也可能导致镜像包含比特定工作负载所需更多的组件。Docker 强化镜像采用最小化设计方法，帮助减少镜像大小、限制攻击面并简化安全和合规工作流程。

强化镜像通过最小化容器镜像中的内容来解决这个问题。更少的软件意味着更少的漏洞、更快的部署，以及每周需要追踪的红色仪表板警告更少。

对于平台工程师和安全团队来说，强化镜像提供了一种摆脱 CVE 分类循环的方法，让您能够专注于交付安全、合规的基础设施，而无需不断地救火。

## 什么是强化镜像？

强化镜像是一种经过刻意精简和安全加固的容器镜像，以减少漏洞并满足严格的安全和合规要求。与可能包含增加风险的非必要组件的标准镜像不同，强化镜像经过精简，仅包含安全运行应用程序所需的内容。

## 强化镜像的优势

- 减少攻击面：通过移除非必要组件，强化镜像限制了攻击者的潜在入口点。
- 改善安全态势：定期更新和漏洞扫描有助于确保强化镜像随时间保持安全。
- 便于合规：包含签名元数据（如 SBOM）支持满足法规和组织合规标准。
- 运营效率：更小的镜像大小带来更快的拉取速度、更低的运行时开销和更少的云资源成本。

## 什么是 Docker 强化镜像？

Docker 强化镜像（DHIs）通过将精简、安全的设计与企业级支持和工具相结合，将强化镜像提升到更高层次。这些镜像以安全为核心构建，持续维护、测试和验证，以满足当今最严格的软件供应链和合规标准。

Docker 强化镜像默认安全、设计精简，并由维护者负责维护，因此您无需操心。

## Docker 强化镜像与通用强化镜像的区别

- SLSA 合规构建：Docker 强化镜像按照 [SLSA Build Level 3](../core-concepts/slsa.md) 构建，确保防篡改、可验证和可审计的构建过程，防止供应链威胁。

- Distroless 方法：与捆绑了完整操作系统（包括 shell、包管理器和调试工具）的传统基础镜像不同，[distroless 镜像](../core-concepts/distroless.md)仅保留运行应用程序所需的最小操作系统组件。通过排除不必要的工具和库，它们可以将攻击面减少高达 95%，并可以改善性能和镜像大小。

- 持续维护：所有 DHIs 都会持续监控和更新，以维持接近零已知可利用 [CVE](../core-concepts/cves.md)，帮助您的团队避免补丁疲劳和意外警报。

- 合规就绪：每个镜像都包含加密签名的元数据：
  - [SBOM](../core-concepts/sbom.md) 显示镜像中包含的内容
  - [VEX 文档](../core-concepts/vex.md) 识别哪些漏洞实际上是可利用的
  - [构建来源证明](../core-concepts/provenance.md) 证明镜像的构建方式和位置

- 注重兼容性的设计：Docker 强化镜像在保持与常见 Linux 发行版兼容的同时提供精简的运行时环境。它们移除了 shell 和包管理器等非必要组件以增强安全性，但保留了基于熟悉发行版标准构建的小型基础层。镜像通常提供 musl libc（基于 Alpine）和 glibc（基于 Debian）两种版本，支持广泛的应用程序兼容性需求。

## 为什么使用 Docker 强化镜像？

Docker 强化镜像（DHIs）默认安全、设计精简，并由维护者负责维护，因此您无需操心。它们提供：

- 让您安心的镜像：超精简和 distroless 设计，DHIs 消除了高达 95% 的传统容器攻击面。
- 不再恐慌补丁：通过持续 CVE 扫描和 SLA 支持的修复，Docker 帮助您领先于威胁。
- 审计就绪的镜像：所有 DHIs 都包含签名的 SBOM、VEX 和来源证明，支持安全和合规工作流程。
- 与您的技术栈兼容的镜像：提供 Alpine 和 Debian 版本，DHIs 可直接放入您现有的 Dockerfile 和管道中。
- 企业支持的镜像：通过 Docker 的支持和对关键漏洞的快速响应获得安心。
